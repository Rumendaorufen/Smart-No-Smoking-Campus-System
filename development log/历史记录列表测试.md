

# 📋 开发技术报告：Web-Agent 对话记忆与安全架构升级

**项目名称**：智慧校园禁烟监控系统 (Smart No-Smoking Campus)
**模块名称**：智能问答代理 (`web-agent`)
**报告日期**：2026-04-22

## 1. 架构演进概述
在本次迭代中，`web-agent` 从一个“无状态的单次 SQL 查询工具”成功升级为**“具备长期记忆、支持多用户隔离、且物理权限绝对安全的 AI 分析引擎”**。

核心解决的痛点包括：
1. 大模型无法联系上下文（如“今天有多少报警？” -> “那昨天呢？”）。
2. 多用户并发时的对话记录隔离问题。
3. LangChain 框架黑盒导致的“记忆丢失” Bug。
4. MySQL 针对大模型 Agent 的最小权限配置盲区。

---

## 2. 数据库架构与权限深度防御 (Security & Storage)

### 2.1 双表分离设计
为了实现前端会话列表与底层消息记录的解耦，我们设计了双表架构：
* **`ai_chat_history` (消息明细表)**：存储具体的对话内容（JSON格式）。由 Python 端通过 `SQLChatMessageHistory` 自动管理。使用 `session_id` 作为外键索引，保证查询速度。
* **`ai_conversations` (会话管理表)**：存储对话标题、所属 `user_id` 和创建时间。由 Java 后端 (`web-back`) 管理，用于前端渲染侧边栏列表。

### 2.2 MySQL 权限“避坑”与配置
在赋予 AI 专用的只读账号（`ai_reader`）权限时，我们排查并解决了一个经典的框架兼容性问题：
* **问题现象**：AI 在查询视图时抛出 `1142: SHOW VIEW command denied`。
* **根本原因**：LangChain `SQLDatabaseToolkit` 需要先通过 `SHOW CREATE TABLE` 了解视图的列结构（Schema），仅仅赋予 `SELECT` 权限不足以支撑 AI 的“探索”行为。
* **最终权限策略**：
    * **业务数据视图**（绝对安全）：仅授予 `SELECT` 和 `SHOW VIEW` 权限，彻底杜绝 AI 删库风险。
    * **历史记录表**（可控写入）：为 `ai_chat_history` 单独授予 `SELECT`, `INSERT` 权限，允许 AI 自动保存用户的聊天记录。

## 3. 核心逻辑重构 (Agent Service Implementation)

### 3.1 跨越 LangChain 的“记忆黑盒”
在使用原生的 `RunnableWithMessageHistory` 或 `MessagesPlaceholder` 与 `create_sql_agent` 结合时，由于内部 AgentExecutor 的封包逻辑问题，导致历史记录（Chat History）经常被静默丢弃。

**解决方案：降维打击的“手动注入法”**
我们废弃了不稳定的中间件，重写了 `AgentService.ask()` 方法，采用了更硬核且 100% 可靠的方案：
1.  **精确读取**：使用 `SQLChatMessageHistory` 根据传入的 `conversation_id` 连接数据库读取历史记录。
2.  **Token 保护**：截取最近 10 条（5轮）对话，防止上下文过长导致大模型 API 计费爆炸或逻辑迷失。
3.  **字符串拼接**：将历史记录直接转化为纯文本，拼接到系统 Prompt 和当前问题中。
4.  **手动持久化**：在 LLM 返回结果后，手动调用 `add_user_message` 和 `add_ai_message` 存入数据库。

这种方案不仅解决了记忆丢失问题，还让发给 LLM 的指令完全透明可控。

### 3.2 配置文件工程化
将硬编码的配置重构为标准的工厂模式：
* 使用 `@dataclass(frozen=True)` 确保配置项在运行时不可变。
* 通过 `dotenv` 自动读取 `.env` 文件，实现 API 密钥和数据库连接串（包含 `charset=utf8mb4` 防乱码设置）的本地化隔离，保障代码库安全。

## 4. 边界划分与下一步计划 (Next Steps)

目前 Python 侧（`web-agent`）的底层逻辑、数据库存取、大模型推理已**全链路跑通**并完成独立测试。

**接下来的系统协作边界已明确：**
1.  **前端 (Vue)**：负责展示对话 UI，用户点击“新建对话”或发送消息。
2.  **Java 中台 (web-back)**：
    * **鉴权与管家**：拦截前端请求，校验 JWT。
    * **生成 ID**：如果是新对话，Java 负责生成 UUID（`conversation_id`）并存入 `ai_conversations` 表。
    * **代理转发**：将用户的 `message` 和生成的 `conversation_id` 打包，通过 HTTP POST 转发给 Python 的 5050 端口。
3.  **Python 代理 (web-agent)**：只认 `conversation_id`，专心做数据库查询和回答生成。
