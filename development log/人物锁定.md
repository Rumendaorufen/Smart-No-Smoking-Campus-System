动态冷却圈 / Dynamic Cooldown Circle 的核心目标是解决**移动吸烟**和**ID跳变**导致的重复报警问题。

以下是该方案的技术要点总结：

### 1. 核心机制：动态锚点 (Dynamic Anchor)

该方案摒弃了将冷却区域固定在“报警发生时的坐标”这一做法，而是将冷却区域**绑定在触发报警的人物 ID 上**。

* **绑定**：当系统确认某人吸烟并触发报警时，创建一个冷却记录，不仅记录坐标和时间，还记录该人物的 `Tracking ID`。
* **跟随**：在随后的每一帧中，只要该 `Tracking ID` 依然存在于画面中，系统就会实时更新冷却圈的中心坐标，使其跟随人物移动。

### 2. 双模态冷却逻辑

为了应对计算机视觉中常见的追踪丢失问题，该方案采用了双模态策略：

* **动态模式 (Dynamic Mode)**：
* **触发条件**：AI 成功追踪到人物 ID。
* **行为**：冷却圈随人物移动。如果该人物走到画面另一端继续吸烟，因为冷却圈一直跟着他，新检测到的烟头依然落在圈内，因此**不会**触发重复报警。


* **静态模式 (Static Mode)**：
* **触发条件**：人物 ID 丢失（如被遮挡、离开画面或追踪器失效）。
* **行为**：冷却圈**立即停留在最后一次检测到的位置**，退化为传统的静态区域冷却。这防止了因 ID 丢失导致的误报（例如 ID 变了，但人还在原地，新 ID 产生的烟头依然会被遗留的静态圈捕获）。



### 3. 数据结构升级

`active_cooldowns` 字典不仅存储简单的坐标，还存储了状态类型：

```python
self.active_cooldowns = {
    key: {
        'pos': (cx, cy),      # 当前冷却圈中心坐标
        'time': timestamp,    # 报警触发时间
        'type': 'dynamic'     # 或 'static'
    }
}

```

* 如果 `key` 是 `int` 类型（Person ID），则视为动态记录，每帧尝试更新 `pos`。
* 如果 `key` 是 `string` 类型（如 UUID），则视为静态记录，`pos` 保持不变。

### 4. 判定流程优化

每一帧检测到疑似烟头时，判定流程如下：

1. **维护**：遍历 `active_cooldowns`，如果对应的 Person ID 还在画面里，更新其 `pos`。
2. **计算**：计算当前烟头坐标与所有冷却圈（无论动态还是静态）的欧氏距离。
3. **抑制**：如果距离小于设定半径（如 200px），则判定为“已知事件”，**抑制报警**。
4. **触发**：如果距离大于半径，则视为“新事件”，触发报警并生成一个新的冷却记录（优先尝试绑定当前最近的人 ID）。

### 5. 解决的痛点

* **移动吸烟**：解决了“边走边抽”导致走出旧冷却圈从而触发新报警的问题。
* **ID 跳变**：即使 ID 发生变化（如从 ID:1 变成 ID:2），只要 ID:1 留下的冷却圈（静态化）依然覆盖该区域，或者新 ID 产生的烟头位置没变，就不会重复报警。
