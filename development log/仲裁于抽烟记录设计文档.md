## 1. 引言 (Introduction)

### 1.1 设计背景

在 AI 视频分析场景中，受光线、遮挡及相似动作（如进食、打电话）影响，模型不可避免会产生误报。为了保障监测数据的权威性与准确性，系统引入 **"Human-in-the-Loop" (人机回环)** 机制。通过人工对 AI 报警结果进行二次仲裁，实现误报剔除、违规确认及证据固化。

### 1.2 核心流程

业务流程分为三个阶段：

1. **AI 检测阶段**：YOLOv8 识别吸烟行为，生成原始报警记录（待审核）。
2. **人工仲裁阶段**：管理员在“审核台”对记录进行快速判断（确认/误报）。
3. **档案管理阶段**：对已确认记录进行归档、查询、导出，对误报记录进行清洗或用于模型再训练。

## 2. 数据库设计 (Database Design)

### 2.1 状态机定义 (State Machine)

为了有效管理数据流转，在 `Alarms` 表中引入生命周期状态码。

| 状态码 (`status`) | 定义 | 含义 | 下一步操作 |
| --- | --- | --- | --- |
| **0** | **Pending (待审核)** | AI 刚生成的原始记录，默认状态 | 等待人工仲裁 |
| **1** | **Confirmed (已确认)** | 经人工核实确为吸烟行为 | 进入档案库，永久保留证据 |
| **2** | **False Positive (误报)** | 经人工核实为误判 | 进入误报库，保留30天用于训练 |
| **9** | **Ignored (已忽略)** | 测试数据或无效数据 | 等待自动清理脚本删除 |

### 2.2 数据表变更 (`Alarms`)

在原有表结构基础上，增加审核相关字段：

```python
class Alarms(db.Model):
    __tablename__ = 'alarms'
    
    # ... 原有字段 (id, device_id, type, roi_url, video_url, confidence, created_at) ...

    # === 新增字段 ===
    status = db.Column(db.Integer, default=0, index=True, comment='0:待审, 1:确认, 2:误报, 9:忽略')
    
    # 审核审计信息
    audit_time = db.Column(db.DateTime, nullable=True, comment='审核时间')
    auditor_id = db.Column(db.Integer, db.ForeignKey('users.id'), nullable=True, comment='审核人ID')
    audit_remark = db.Column(db.String(255), nullable=True, comment='审核备注/驳回原因')

```

## 3. API 接口设计 (API Design)

### 3.1 获取审核列表 (Audit Console)

* **URL**: `GET /api/v1/alerts/pending`
* **描述**: 获取所有状态为 `0` (待审核) 的报警记录。
* **参数**: `page`, `page_size`
* **响应**:
```json
{
  "data": [
    {
      "id": 101,
      "device_name": "男寝一号楼-东侧",
      "roi_url": "/static/evidence/xxx.jpg",
      "video_url": "/static/evidence/xxx.mp4",
      "confidence": 0.85,
      "created_at": "2026-01-16 14:30:00"
    }
  ]
}

```



### 3.2 提交审核结果 (Submit Audit)

* **URL**: `POST /api/v1/alerts/<id>/audit`
* **描述**: 提交人工判断结果。
* **Body**:
```json
{
  "status": 1,          // 1=确认, 2=误报
  "remark": "确实在抽烟" // 可选
}

```



### 3.3 历史档案查询 (Archive Search)

* **URL**: `GET /api/v1/alerts/archive`
* **描述**: 多维度查询已归档数据（状态不为0的）。
* **参数**:
* `start_time`, `end_time` (时间范围)
* `device_id` (设备筛选)
* `status` (筛选 1-已确认 或 2-误报)



### 3.4 导出报表 (Export)

* **URL**: `GET /api/v1/alerts/export`
* **描述**: 将筛选后的记录导出为 Excel 文件。

## 4. 前端界面设计 (UI/UX Design)

### 4.1 页面一：报警仲裁台 (Audit Console)

**设计目标**：极速处理，减少管理员操作路径。

* **布局模式**：
* **左侧 (25%)**：待办列表（显示简略信息：时间、地点、置信度）。
* **右侧 (75%)**：沉浸式详情审核区。


* **核心组件**：
* **证据大图**：使用 `el-image`，默认显示检测框。支持滚轮缩放。
* **GIF 预览**：视频区域鼠标悬停时自动播放，移开停止（类似 YouTube 预览）。
* **置信度红绿灯**：
* Conf > 0.8: 显示绿色标签 "高置信度"
* Conf < 0.5: 显示红色标签 "需人工复核"




* **快捷交互**：
* 键盘 `←` / `→`：切换上一条/下一条。
* 键盘 `Y` 或 `Enter`：**确认违规** (自动跳转下一条)。
* 键盘 `N` 或 `Del`：**标记误报** (自动跳转下一条)。



### 4.2 页面二：违规记录档案 (Records Archive)

**设计目标**：查询、追溯、管理。

* **布局模式**：标准的 Admin 表格布局。
* **顶部筛选栏**：时间选择器、设备下拉框、状态单选框（全部/已确认/误报）。
* **数据表格 (`el-table`)**：
* **缩略图列**：点击可查看大图/视频弹窗。
* **状态列**：使用 `el-tag` 区分颜色（绿色=已确认，灰色=误报）。
* **操作列**：
* [删除]：物理删除文件 + 软删除记录。
* [加入负样本]：仅针对“误报”数据，标记该图片用于下次训练。




* **底部功能区**：
* `[导出 Excel]` 按钮。
* `[批量删除]` 按钮。

## 5. 存储维护与自动清理 (Storage Strategy)

为了防止长期运行导致服务器磁盘爆满，实施以下**分级存储策略**：

### 5.1 自动清理任务 (APScheduler)

后端配置每日凌晨 3:00 执行的定时任务。

```python
def cleanup_job():
    # 策略 1: 清理 "误报 (Status=2)" 和 "忽略 (Status=9)" 的数据
    # 保留周期: 30天 (留作训练素材，之后删除)
    deadline = datetime.now() - timedelta(days=30)
    
    useless_alarms = Alarms.query.filter(
        Alarms.status.in_([2, 9]), 
        Alarms.created_at < deadline
    ).all()
    
    for alarm in useless_alarms:
        # 1. 物理删除文件 (图片 & 视频)
        os.remove(alarm.roi_path)
        os.remove(alarm.video_path)
        
        # 2. 数据库硬删除
        db.session.delete(alarm)
        
    db.session.commit()
    print(f"🧹 自动清理完成，删除了 {len(useless_alarms)} 条无效记录")

```

### 5.2 证据保留

* **状态=1 (已确认)**：视为违规证据，**默认永久保留**。
* **手动删除**：只有管理员在“档案管理”页面点击删除时，才会被清理。

## 7. 总结

本设计方案通过引入**状态机**和**分级管理**，将单纯的“查看记录”升级为一套完整的**闭环管理系统**。

1. **对于管理员**：快捷键和极速模式极大降低了审核几十条报警的疲劳感。
2. **对于系统维护**：自动清理策略解决了磁盘空间焦虑。
3. **对于 AI 算法**：误报数据的结构化收集，为未来提升模型精度积累了宝贵的“错题集”。
