# 智慧校园禁烟监控系统 — 可行性分析报告

| 项目名称 | 智慧校园禁烟监控系统 (Smart No-Smoking Campus System) |
| --- | --- |
| **文档版本** | V 3.0（与实际系统一致） |
| **拟制日期** | 2026-05-08 |
| **项目性质** | 智慧校园安防升级、计算机视觉 + 大模型智能问答 |
| **核心技术** | YOLOv8 双模型端到端检测 + LangChain SQL Agent + 四服务协同架构 |

---

## 1. 项目背景与建设必要性

### 1.1 现状与痛点

高校禁烟管理长期面临四项核心挑战：

1. **监管盲区多：** 校园面积大，宿舍走廊、消防楼梯间等是吸烟重灾区，安保力量无法全天候覆盖。
2. **取证困难：** 吸烟行为瞬时隐蔽，发现时烟头往往已处理，缺乏直接视频证据，无法有效教育或追责。
3. **火灾隐患大：** 乱扔烟头是高校火灾的主要诱因，传统烟感报警器对室外或微量烟雾反应迟钝。
4. **数据孤岛：** 即便有视频监控，报警数据也缺乏有效分析手段。管理人员难以回答"哪栋楼最严重？""本周趋势如何？"等业务问题。

### 1.2 建设目标

本项目构建一套完整的"视觉监控 + 业务管理 + 智能问答"三位一体平台：

- **AI 视觉引擎：** 7×24 小时自动分析 RTSP 视频流，识别吸烟行为并生成带框证据
- **业务中台：** 统一鉴权、设备管理、报警仲裁、档案归档、系统监控
- **AI 数据分析助手：** 基于大语言模型的自然语言查询，自动生成 SQL 并返回分析报告

---

## 2. 技术可行性分析

### 2.1 整体架构

系统采用 **四服务协同架构**：Vue 3 前端（5173 端口）→ Java Spring Boot 中台（8080）→ Python AI 引擎（5000）+ LangChain AI Agent（5050），MySQL 8.0 共享数据库。

### 2.2 AI 检测算法（web-flask）

采用 **端到端双模型级联检测**，替代旧有的 Pose+CNN 两阶段方案：

1. **Stage 1 — 人员检测：** 使用 YOLOv8s 模型全局找人，ByteTrack 算法追踪人员 ID
2. **Stage 2 — 烟头精检：** 裁切人物上半身 ROI（含 30% 外扩），Batch 打包送入自定义 `best.pt` 烟头模型
3. **FP16 半精度 + GPU 加速：** 推理延迟 ~10ms/帧，单张 RTX 3060 稳定承载多路并发
4. **惯性追踪防闪烁：** 3 帧记忆库相对坐标补全，15 帧连续确认才触发报警
5. **动态冷却圈：** 冷却区域绑定人员 ID 而非固定坐标，解决移动吸烟与 ID 跳变导致的重复报警

**技术优势：** 端到端检测消除了 Pose 方案的累积误差，4000+ 训练样本（3:1 正负比）将误报率从 30% 降至 <1%。

### 2.3 AI 数据分析助手（web-agent）

基于 LangChain `SQLDatabaseToolkit` + OpenAI 兼容大模型：

- 自然语言转 SQL，自动查询 4 个专用视图
- 三层安全防御：视图层隔离 + MySQL 账号权限限制 + Prompt 约束
- 流式 SSE 输出，Markdown 表格渲染
- 多会话管理，对话历史持久化

### 2.4 Java 业务中台（web-back）

- Spring Boot 3 + MyBatis-Plus 提供 RESTful API
- Spring Security + JWT 实现无状态认证
- WebSocket (STOMP/SockJS) 实时推送报警
- SSE 代理透传 AI Agent 流式响应

### 2.5 前端大屏（web-vue）

- Vue 3 Composition API + TypeScript + Element Plus + Pinia + ECharts
- 7 个核心页面覆盖监控、仲裁、档案、管控、对话全流程

## 3. 经济可行性分析

### 3.1 硬件成本

- **利旧现有摄像头：** 直接接入校园 RTSP 监控网络，无需新增摄像头或布线
- **单台服务器承载：** 配备中端显卡（如 RTX 3060，6GB 显存）的单台 PC 即可承载 4-8 路并发 AI 推理
- **存储策略：** 误报记录 30 天自动清理，已确认违规永久保留，磁盘占用可控

### 3.2 软件成本

- **全部使用开源技术栈：** Python、Java、Vue、MySQL、LangChain 均免费
- **大模型 API 按量付费：** GPT-4o-mini 级别的模型单次查询成本极低（< 0.01 元），且可对接任何 OpenAI 兼容 API

### 3.3 人力成本

- 系统自动化运行，安保人员只需处理报警仲裁（人工终审），日均审核量可控
- 无需专职 AI 工程师驻场运维

---

## 4. 操作可行性分析

- **B/S 架构，浏览器访问：** 无需安装客户端软件
- **角色权限分离：** 管理员（全权限）vs 操作员（审核+查看），匹配实际组织分工
- **自然语言数据查询：** AI 助手支持口语化提问（如"最近一周报警趋势"），无需 SQL 技能
- **人工终审兜底：** AI 报警仅为辅助建议，最终确认权始终在人工

---

## 5. 合规性与隐私

| 区域类型 | 监测策略 |
| --- | --- |
| 宿舍走廊、消防楼梯、天台、室外公共区 | 全画面监测，不设隐私遮罩 |
| 宿舍内部、卫生间、淋浴房、更衣室 | 物理隔离，严禁接入 |

- 所有视频数据仅在校园内网传输与存储，不上传公网
- 数据库采用最小权限原则（AI Agent 仅能读取 4 个视图，禁止写操作）

---

## 6. 风险评估与应对

| 风险 | 等级 | 应对策略 |
| --- | --- | --- |
| 夜间/低光环境检测率下降 | 中 | 确保摄像头开启红外夜视；持续收集低光照样本微调模型 |
| 误报（签字笔、棒棒糖） | 低 | 4000+ 训练样本含 1000 负样本，误报率 < 1%；人工终审兜底 |
| RTSP 信号中断 | 中 | 看门狗线程 5 秒超时自动重连；FFmpeg 单线程解码防死锁 |
| GPU 显存不足 | 低 | 全局单例模式 + FP16 半精度，多路并发仅加载一份模型 |
| LLM 被诱导生成恶意 SQL | 低 | 物理层权限拦截（只读账号）+ 视图隔离 + Prompt 约束 |

---

## 7. 结论

项目技术方案经过实际开发验证，四服务架构运行稳定。YOLOv8 双模型级联检测在精度和性能上均满足生产要求，LangChain SQL Agent 有效填补了"数据查询最后一公里"的空白。系统在经济成本、操作门槛、合规性方面均可行。

**结论：项目已完成开发并通过多轮测试验证，具备正式部署条件。**
